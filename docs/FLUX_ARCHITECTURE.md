# FluxCD Architecture Blueprint

This document describes the FluxCD directory structure, reconciliation flow, and
packaging patterns used in the LibrePod Marketplace. It is written as a
reproducible blueprint — you can replicate this same structure in another
project by following the patterns described here.

## Directory Structure

```
repo-root/
├── clusters/                        # FluxCD entry points (per-cluster)
│   ├── <cluster-name>/              # One directory per cluster
│   │   ├── flux-system/             # Flux bootstrap components
│   │   │   ├── gotk-components.yaml # (auto-generated by flux bootstrap)
│   │   │   ├── gotk-sync.yaml       # (auto-generated by flux bootstrap)
│   │   │   └── kustomization.yaml   # References the two files above
│   │   ├── system-apps.yaml         # Flux Kustomization → infrastructure/system-apps
│   │   ├── system-configs.yaml      # Flux Kustomization → infrastructure/system-configs
│   │   └── user-apps-source.yaml    # Flux Kustomization → infrastructure/user-apps-source
│   └── <another-cluster>/           # Same structure, different variables
│       └── ...
├── infrastructure/                  # Shared infrastructure definitions
│   ├── system-apps/                 # One YAML per system app
│   │   ├── kustomization.yaml       # Lists all app YAML files
│   │   ├── traefik.yaml             # OCIRepository + Flux Kustomization
│   │   ├── casdoor.yaml             # OCIRepository + Flux Kustomization
│   │   └── ...                      # One file per system app
│   ├── system-configs/              # Cluster-wide configuration
│   │   ├── kustomization.yaml       # Lists all config files
│   │   ├── coredns-custom.yaml      # DNS overrides
│   │   ├── traefik-tlsstore.yaml    # TLS certificate store
│   │   ├── cosign-pub.yaml          # Cosign public key for verification
│   │   └── flux-ingressroute.yaml   # Flux dashboard ingress
│   └── user-apps-source/            # Dynamic user app source
│       ├── kustomization.yaml
│       ├── gitrepository.yaml       # GitRepository pointing to on-cluster Gogs
│       └── user-apps.yaml           # Flux Kustomization watching Gogs repo
├── apps/                            # Individual app manifests (OCI artifacts)
│   └── <app-name>/                  # Example: casdoor
│       ├── metadata.yaml            # App metadata for catalog/marketplace
│       ├── base/                    # Environment-agnostic manifests
│       │   ├── kustomization.yaml   # Lists base resources
│       │   ├── namespace.yaml       # Namespace definition
│       │   ├── ocirepository.yaml   # Upstream helm chart OCI source
│       │   ├── helmrelease.yaml     # Flux HelmRelease with defaults
│       │   └── pvc.yaml             # Persistent volume claims
│       └── overlays/
│           └── <overlay-name>/      # Example: librepod
│               ├── kustomization.yaml       # References ../../base + overlay files
│               ├── ingressroute.yaml        # Ingress routing (uses ${BASE_DOMAIN})
│               ├── patch-helmrelease.yaml   # Strategic merge patch on HelmRelease
│               └── patch-storage-class.yaml # Patch to set storageClassName
├── .github/workflows/               # CI pipelines
│   ├── publish-bootstrap.yaml       # Packs clusters/ + infrastructure/ → OCI
│   ├── publish-apps.yaml            # Packs each apps/<name>/ → OCI
│   └── publish-catalog.yaml         # Packs catalog.yaml → OCI
├── catalog.yaml                     # Generated app catalog (auto-generated by CI)
├── cosign.pub                       # Public key for artifact verification
└── scripts/
    └── generate-catalog.sh          # Generates catalog.yaml from metadata.yaml files
```

## Three-Layer Separation

The architecture separates concerns into three layers:

| Layer | Directory | Responsibility |
|-------|-----------|----------------|
| **Cluster entry points** | `clusters/<name>/` | Defines *which paths to reconcile* and per-cluster variables |
| **Infrastructure wiring** | `infrastructure/` | Defines *what sources to watch* and dependency ordering |
| **App manifests** | `apps/<name>/` | Contains *the actual manifests*, packaged as OCI artifacts |

**Adding a new cluster** = copy `clusters/<name>/` with different `BASE_DOMAIN`.
**Adding a new system app** = add a directory under `apps/` + a single YAML under `infrastructure/system-apps/`.

## Reconciliation Flow

```
1. BOOTSTRAP (one-time, via helm install)
   ┌──────────────────────────────────────────────────────┐
   │ flux-instance Helm chart creates:                     │
   │   OCIRepository: librepod-bootstrap                   │
   │     url: oci://ghcr.io/<owner>/marketplace/bootstrap  │
   │     contains: clusters/ + infrastructure/             │
   └────────────────────┬─────────────────────────────────┘
                        │ Flux reads clusters/<cluster-name>/
                        ▼
2. CLUSTER-LEVEL KUSTOMIZATIONS (clusters/<name>/*.yaml)
   ┌──────────────────────┐
   │ system-apps.yaml     │──→ ./infrastructure/system-apps
   │ system-configs.yaml  │──→ ./infrastructure/system-configs
   │ user-apps-source.yaml│──→ ./infrastructure/user-apps-source
   └──────────────────────┘
   All sourceRef → OCIRepository "librepod-bootstrap"
   All use postBuild.substitute to inject per-cluster variables

                        ▼
3. INFRASTRUCTURE LAYER (infrastructure/system-apps/<app>.yaml)
   Each file defines an OCIRepository + Kustomization per app:
   ┌─────────────────────────────────────────────────┐
   │ OCIRepository: <app-name>                        │
   │   url: oci://ghcr.io/<owner>/marketplace/apps/<app> │
   │   tag: "1.0.0"                                   │
   │   verify: cosign                                 │
   │                                                  │
   │ Kustomization: <app-name>                        │
   │   sourceRef: OCIRepository "<app-name>"          │
   │   path: ./overlays/<overlay-name>                │
   │   dependsOn: [other-system-apps...]              │
   └─────────────────────────────────────────────────┘

                        ▼
4. APP LAYER (apps/<name>/ — packaged as OCI artifact)
   ┌────────────────────────────────────────┐
   │ base/  — environment-agnostic          │
   │   namespace.yaml                       │
   │   ocirepository.yaml (helm chart)      │
   │   helmrelease.yaml                     │
   │   pvc.yaml                             │
   │ overlays/<overlay-name>/ — overlay     │
   │   ingressroute.yaml (${BASE_DOMAIN})   │
   │   patch-helmrelease.yaml               │
   │   patch-storage-class.yaml             │
   └────────────────────────────────────────┘
```

## File-by-File Breakdown (Casdoor Example)

### Cluster Entry Point — `clusters/librepod-dev/system-apps.yaml`

```yaml
apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata:
  name: system-apps
  namespace: flux-system
spec:
  interval: 1h
  retryInterval: 2m
  timeout: 5m
  sourceRef:
    kind: OCIRepository
    name: librepod-bootstrap     # The bootstrap OCI artifact
  path: ./infrastructure/system-apps  # Where to find manifests
  prune: true
  postBuild:
    substitute:
      BASE_DOMAIN: "librepod.dev"     # Per-cluster variable
```

**Purpose:** Tells Flux to look at `infrastructure/system-apps/` inside the
bootstrap artifact. There is one of these per "layer" (system-apps,
system-configs, user-apps-source), and the set is duplicated per cluster.

### Infrastructure App Definition — `infrastructure/system-apps/casdoor.yaml`

```yaml
# Part 1: WHERE to get the app manifests
apiVersion: source.toolkit.fluxcd.io/v1
kind: OCIRepository
metadata:
  name: casdoor
  namespace: flux-system
spec:
  interval: 10m
  url: oci://ghcr.io/librepod/marketplace/apps/casdoor
  ref:
    tag: "1.0.0"
  verify:
    provider: cosign
    secretRef:
      name: cosign-pub
---
# Part 2: HOW to deploy it
apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata:
  name: casdoor
  namespace: flux-system
spec:
  dependsOn:
    - name: nfs-provisioner
    - name: traefik
  interval: 1h
  retryInterval: 2m
  timeout: 5m
  sourceRef:
    kind: OCIRepository
    name: casdoor
  path: ./overlays/librepod
  prune: true
  wait: true
  postBuild:
    substitute:
      BASE_DOMAIN: "${BASE_DOMAIN}"  # Inherited from parent Kustomization
```

**Purpose:** A single file wiring the OCI artifact to the deployment.
`dependsOn` controls ordering. Each system app has one of these files, all
listed in `system-apps/kustomization.yaml`.

### App Base — `apps/casdoor/base/kustomization.yaml`

```yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization

namespace: casdoor

resources:
- namespace.yaml
- ocirepository.yaml
- helmrelease.yaml
- pvc.yaml
```

**Purpose:** Lists environment-agnostic resources. The `namespace` field sets
the default namespace for all resources.

### App Base — `apps/casdoor/base/ocirepository.yaml`

```yaml
apiVersion: source.toolkit.fluxcd.io/v1
kind: OCIRepository
metadata:
  name: casdoor-helm-charts
spec:
  interval: 24h
  url: oci://registry-1.docker.io/casbin/casdoor-helm-charts
  ref:
    semver: "2.353.*"
```

**Purpose:** Points to the upstream helm chart. Note the `semver` range — Flux
automatically resolves to the latest matching version.

### App Base — `apps/casdoor/base/helmrelease.yaml`

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: casdoor
spec:
  interval: 12h
  chartRef:
    kind: OCIRepository
    name: casdoor-helm-charts
  values:
    config: |
      appname = casdoor
      httpport = 8000
      ...
```

**Purpose:** Flux HelmRelease referencing the helm chart OCI repository. All
default values are baked in — no environment variables needed.

### App Overlay — `apps/casdoor/overlays/librepod/kustomization.yaml`

```yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization

namespace: casdoor

resources:
- ../../base
- ingressroute.yaml

configMapGenerator:
- name: init-data
  files:
    - ./init_data.json
  options:
    disableNameSuffixHash: true

patches:
- path: ./patch-helmrelease.yaml
  target:
    kind: HelmRelease
- path: ./patch-storage-class.yaml
  target:
    kind: PersistentVolumeClaim
```

**Purpose:** Composes base + overlay. Adds ingress routing, configmap
generators, and strategic merge patches for environment-specific config.

### App Overlay — `apps/casdoor/overlays/librepod/ingressroute.yaml`

```yaml
apiVersion: traefik.io/v1alpha1
kind: IngressRoute
metadata:
  name: casdoor
spec:
  entryPoints:
  - web
  - websecure
  routes:
  - kind: Rule
    match: Host(`casdoor.${BASE_DOMAIN:=libre.pod}`)
    services:
    - name: casdoor
      port: 8000
```

**Purpose:** Uses `${BASE_DOMAIN}` variable substitution (via Flux postBuild).
The `:=libre.pod` provides a default fallback.

### App Overlay — `apps/casdoor/overlays/librepod/patch-storage-class.yaml`

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: not-important
spec:
  storageClassName: nfs-client
```

**Purpose:** Strategic merge patch that sets `storageClassName` on all PVCs in
the base. The `name: not-important` is required by kustomize but is ignored —
the patch targets by `kind: PersistentVolumeClaim`.

### App Metadata — `apps/casdoor/metadata.yaml`

```yaml
apiVersion: marketplace/v1
kind: AppDefinition
metadata:
  name: casdoor
spec:
  displayName: "Casdoor"
  description: "Open-source IAM / SSO platform"
  icon: "https://..."
  category: "Security"
  version: "2.353.0"

  source:
    type: oci-kustomize
    url: "oci://ghcr.io/librepod/marketplace/apps/casdoor"
    path: ./overlays/librepod

  params:
    required:
      - name: BASE_DOMAIN
        description: "Base domain for cluster"
        type: string

  dependencies:
    required:
      - kind: IngressController
        description: "Traefik"
      - kind: StorageClass
        description: "nfs-client"

  templates:
    source: |
      # YAML template for generating OCIRepository dynamically
    release: |
      # YAML template for generating Kustomization dynamically
    kustomization: |
      # YAML template for composing source + release
```

**Purpose:** Used by the catalog generator and marketplace UI. Defines app
metadata, version, dependencies, and templates for dynamic resource generation.

## Dependency Chain

```
librepod-bootstrap OCI artifact (clusters/ + infrastructure/)
│
├── system-apps Kustomization
│   ├── nfs-provisioner (independent)
│   ├── step-certificates (independent)
│   ├── traefik (independent)
│   ├── reflector (independent)
│   ├── cert-manager (dependsOn: traefik)
│   ├── step-issuer (dependsOn: step-certificates)
│   ├── gogs (dependsOn: nfs-provisioner + traefik)
│   ├── casdoor (dependsOn: nfs-provisioner + traefik)
│   ├── oauth2-proxy (dependsOn: traefik)
│   ├── wg-easy (dependsOn: traefik)
│   └── whoami (dependsOn: traefik)
│
├── system-configs Kustomization (dependsOn: system-apps)
│   ├── coredns-custom
│   ├── traefik-tlsstore
│   ├── flux-ingressroute
│   └── cosign-pub
│

```

## Two OCI Repositories Per App

Each app involves two separate OCI repositories:

1. **App artifact** (`oci://ghcr.io/<owner>/marketplace/apps/<name>`)
   Contains the kustomize manifests (base + overlays). Published by the
   `publish-apps.yaml` CI workflow.

2. **Helm chart** (upstream, e.g. `oci://registry-1.docker.io/...`)
   The upstream chart referenced from `base/ocirepository.yaml`.

The app artifact is the "packaging wrapper" (namespace, PVC, ingress, patches)
around the upstream chart. This pattern lets you bake in all defaults and
environment-specific config into a versioned, signed OCI artifact.

## CI/CD Pipeline

Three GitHub Actions workflows handle OCI artifact publishing:

| Workflow | Trigger | What it publishes | OCI path |
|----------|---------|-------------------|----------|
| `publish-bootstrap.yaml` | Push to master (changes in `clusters/` or `infrastructure/`) | `clusters/` + `infrastructure/` | `oci://ghcr.io/<owner>/marketplace/bootstrap` |
| `publish-apps.yaml` | Push to master (changes in `apps/*/base/`, `overlays/`, `metadata.yaml`) | Each `apps/<name>/` individually | `oci://ghcr.io/<owner>/marketplace/apps/<name>` |
| `publish-catalog.yaml` | After publish-apps completes, or push to `metadata.yaml` | `catalog.yaml` | `oci://ghcr.io/<owner>/marketplace/catalog` |

All three use `flux push artifact` to publish and `cosign sign` to sign both
versioned and `latest` tags.

### publish-bootstrap.yaml

Packs `clusters/` and `infrastructure/` into a single OCI artifact. Version is
derived from git tag or commit SHA.

### publish-apps.yaml

Detects which apps changed (via git diff) and publishes only those. Supports
manual dispatch with `["all"]` to republish everything. Reads version from
`metadata.yaml`.

### publish-catalog.yaml

Runs the catalog generator script, commits `catalog.yaml` if changed, then
publishes it as an OCI artifact.

## Variable Substitution

Flux's `postBuild.substitute` mechanism injects variables into manifests:

- **Cluster level** (`clusters/<name>/system-apps.yaml`): sets `BASE_DOMAIN`
  per cluster (e.g., `"librepod.dev"` vs `"my-server.com"`)
- **Infrastructure level** (`infrastructure/system-apps/casdoor.yaml`): passes
  `"${BASE_DOMAIN}"` through — inherited from parent
- **Overlay level** (`apps/casdoor/overlays/librepod/ingressroute.yaml`):
  uses `${BASE_DOMAIN:=libre.pod}` with a fallback default

## Cosign Verification

Every OCI artifact is signed with cosign. The cluster verifies signatures
before deploying:

1. CI signs artifacts with a private key (`COSIGN_PRIVATE_KEY` secret)
2. The public key is stored as a ConfigMap in `system-configs/cosign-pub.yaml`
3. Each `infrastructure/system-apps/<app>.yaml` includes `verify.provider: cosign`

## Replication Checklist

To replicate this architecture in a new project:

1. **Create the directory structure:**
   ```
   clusters/<cluster-name>/
   infrastructure/system-apps/
   infrastructure/system-configs/
   infrastructure/user-apps-source/  (optional, for dynamic user apps)
   apps/<app-name>/base/
   apps/<app-name>/overlays/<env>/
   ```

2. **Generate cosign key pair:**
   ```bash
   cosign generate-key-pair
   # Save cosign.pub in infrastructure/system-configs/cosign-pub.yaml
   # Add COSIGN_PRIVATE_KEY and COSIGN_PASSWORD to GitHub secrets
   ```

3. **Create cluster entry points** (`clusters/<name>/system-apps.yaml` etc.)
   with `sourceRef` pointing to the bootstrap OCI artifact

4. **Create infrastructure wiring** for each system app — a single YAML with
   OCIRepository + Kustomization + dependsOn

5. **Package each app** using the base/overlay kustomize pattern

6. **Set up CI workflows:**
   - `publish-bootstrap.yaml` — packs clusters/ + infrastructure/
   - `publish-apps.yaml` — packs each app directory
   - `publish-catalog.yaml` — packs the generated catalog (optional)

7. **Bootstrap the cluster:**
   ```bash
   helm install flux-operator oci://ghcr.io/controlplaneio-fluxcd/charts/flux-operator \
     --namespace flux-system --create-namespace

   helm install flux-instance oci://ghcr.io/controlplaneio-fluxcd/charts/flux-instance \
     --namespace flux-system \
     --set instance.sync.kind=OCIRepository \
     --set instance.sync.url=oci://ghcr.io/<owner>/marketplace/bootstrap \
     --set instance.sync.path=./clusters/<cluster-name>
   ```
